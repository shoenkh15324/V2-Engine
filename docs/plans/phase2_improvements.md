## 1. 개요

### 1.1 목적

현재 V2-Engine 시스템의 성능 병목 현상을 체계적으로 분석하고, 우선순위에 따라 개선하는 것이 목표입니다. 이 단계에서는 다음을 다룹니다:

- **메일박스 변형**: 뮤텍스 경쟁 해결 (가장 높은 우선순위)
- **스케줄링 개선**: 워커 간 로드 불균형 해결
- **배치 처리 최적화**: 런타임 적응적 배치
- **메트릭 개선**: 오버헤드 감소 및 유용성 향상
- **메시지 라우팅**: 불필요한 오버헤드 제거

### 1.2 개선 사항 요약

| 구분 | 개선 사항 | 예상 효과 | 우선순위 |
|------|----------|----------|----------|
| **메일박스** | MPSC 잠금 해제 메일박스 | 처리량 2-3배 향상 | **치명적** |
| **스케줄링** | 작업 도둑질 스케줄러 | 워커 간 로드 균형 | **높음** |
| **배치 처리** | 적응적 maxBatch 조절 | 다양한 워크로드 최적화 | **중간** |
| **메트릭** | 잠금 해제 카운터, 배치 수집 | 측정 오버헤드 감소 | **중간** |
| **라우팅** | 이중 뮤텍스 제거, 핫 패스 최적화 | 오버헤드 50% 감소 | **높음** |

## 2. 현재 시스템 분석

### 2.1 발견된 병목 현상

| 병목 현상 | 위치 | 영향 | 우선순위 | 설명 |
|------------|----------|--------|----------|------|
| **메일박스 뮤텍스 경쟁** | `mailbox.hpp` push/pop/empty/count | 주요 throughput 제한 | **치명적** | 64개 워커가 16개 액터의 뮤텍스를 경쟁 |
| **이중 뮤텍스 획득** | `actor_context.cpp:26` | 불필요한 오버헤드 | **높음** | enqueue()에서 push() 후 count() 호출로 이중 잠금 |
| **배치 루프당 메시지별 뮤텍스** | `actor_context.cpp:37` | maxBatch에 비례 | **높음** | while 루프에서 매번 pop() 호출하며 뮤텍스 획득 |
| **64 워커 > 16 액터 불일치** | 벤치마크 설정 | 과도한 세마포어 경쟁 | **중간** | 워커 수가 액터 수보다 많으면 비효율적 |
| **Time::now() 핫 패스** | `actor_context.cpp:33,41` | ~12ms overhead/100K | **낮음** | 핫 패스에서 시각 호출 발생 |
| **작업 도둑질 없음** | 워커 설계 | 로드 불균형 가능 | **낮음** | 특정 워커가 과부하일 때 다른 워커가 유휴 |


## 3. 메일박스 변형 구현

### 3.1 목표

1. `IMailbox<T>` 인터페이스 추출
2. 현재 `Mailbox<T>`를 `MutexMailbox<T>`로 이름 변경
3. `MPSCMailbox<T>` 잠금 해제 구현
4. `ActorContext`를 인터페이스 기반으로 리팩토링

### 3.2 현재 메일박스 분석

**파일**: `src/core/actor_system/runtime/mailbox.hpp`

현재 메일박스는 `std::mutex`로 보호되는 바운드 링 버퍼입니다. 동시성 패턴은:
- **프로듀서 (push)**: MPSC - 여러 스레드에서 동시 전송
- **컨슈머 (pop)**: SC - 단일 워커 스레드만 읽음

### 3.3 IMailbox 인터페이스

**새 파일**: `src/core/actor_system/runtime/i_mailbox.hpp`

```cpp
template <typename T>
class IMailbox {
public:
    virtual ~IMailbox() = default;
    virtual bool push(T&& msg) = 0;
    virtual bool pop(T& out) = 0;
    virtual bool empty() const = 0;
    virtual size_t count() const = 0;
    virtual size_t capacity() const = 0;
};
```

### 3.4 MPSCMailbox 설계

Dmitry Vyukov의 바운드 MPMC 큐를 MPSC용으로 단순화:

- **프로듀서**: `fetch_add` 또는 CAS로 슬롯 확보
- **컨슈머**: `tail_`을 통해 단독으로 읽음
- **시퀀스 카운터**: 안전한 슬롯 접근 보장
- **캐시 라인 패딩**: 거짓 공유 방지 (`alignas(64)`)

### 3.5 메모리 순서

| 연산 | 순서 | 이유 |
|-----------|-------|--------|
| `head_.fetch_add` / CAS | relaxed | 원자성만 필요 |
| push의 `tail_.load` | acquire | 컨슈머의 tail 전진 확인 |
| push의 `sequence.store` | release | 컨슈머에게 데이터 게시 |
| pop의 `sequence.load` | acquire | 프로듀서의 데이터 확인 |
| pop의 `tail_.store` | release | 프로듀서에게 슬롯 가용 신호 |

### 3.6 예상 효과

- 높은 경쟁 환경 (64 워커, 16 액터)에서 **2-3배 처리량 개선**
- 뮤텍스 오버헤드 제거로 **지연 시간 감소**

## 4. 작업 도둑질 스케줄러

### 4.1 문제

현재 워커는 고정된 액터에 바인딩되지 않지만, 스케줄링이 불균형할 수 있습니다:
- 특정 워커가 여러 액터의 메시지를 처리하는 동안 다른 워커는 유휴
- 세마포어 기반 디스패치가 로드 균형을 보장하지 않음

### 4.2 해결책

**작업 도둑질 (Work Stealing)** 패턴 구현:

1. 각 워커有自己的 대기 큐 (ready queue)
2. 디스패처가 작업을 한 워커에만 디스패치
3. 유휴 워커가 다른 워커의 큐에서 "도둑질"

### 4.3 설계

```
현재:
Dispatcher ──> 세마포어 ──> Worker 0 (항상 같은 큐)
                        ──> Worker 1
                        ──> Worker N

개선:
Dispatcher ──> Worker 0 큐 ──> Worker 0
           ──> Worker 1 큐 ──> Worker 1 (유휴 시 Worker 0 큐에서 도둑질)
           ──> Worker N 큐 ──> Worker N
```

### 4.4 구현 고려 사항

- **도둑질 전략**: 랜덤 선택 vs 가장 많이 가진 워커 선택
- **동기화**: 워커 간 큐 접근 시 레이스 컨디션 방지
- **오버헤드**: 도둑질 자체의 비용 vs 이점 비교

### 4.5 예상 효과

- 워커 간 **로드 균형 향상**
- 특정 워커의 **과부하 방지**
- 전체 **처리량 향상** (예: 10-20%)

## 5. 동적 워커 조절

### 5.1 문제

현재 워커 수는 생성 시 고정됩니다:
- 적은 부하일 때 많은 워커는 리소스 낭비
- 많은 부하일 때 적은 워커는 병목

### 5.2 해결책

**동적 워커 조절 (Dynamic Worker Scaling)**:

1. 시스템 부하에 따라 워커 수를 동적으로 조절
2. CPU 사용률 기반 또는 메시지 큐 깊이 기반
3. 미리 워커 풀을 생성해두고 활성화/비활성화

### 5.3 설계

```cpp
class AdaptiveWorkerPool {
    std::vector<std::thread> workers;
    std::atomic<int> activeCount;
    std::atomic<double> loadFactor;  // 0.0 ~ 1.0

    void adjustWorkers() {
        double load = calculateLoad();
        if (load > 0.8 && activeCount < maxWorkers) {
            activateWorker();
        } else if (load < 0.3 && activeCount > minWorkers) {
            deactivateWorker();
        }
    }
};
```

### 5.4 로드 측정 지표

| 지표 | 측정 방법 | 임계값 |
|------|----------|--------|
| 메일박스 깊이 | `mailbox.count()` | > 용량의 80% |
| CPU 사용률 | `/proc/stat` 또는 `getrusage` | > 80% |
| 처리 대기열 | 디스패처 준비 큐 크기 | > 워커 수 * 2 |

### 5.5 예상 효과

- **리소스 효율성 향상**: 적은 부하 시 워커 수 감소
- **응답성 향상**: 많은 부하 시 워커 수 증가
- **전력 소비 절감**: 유휴 시 워커 비활성화

## 6. 적응적 배치 처리

### 6.1 문제

현재 `maxBatch`는 설정 시 고정됩니다:
- 작은 배치: 스케줄링 오버헤드 증가
- 큰 배치: 지연 시간 증가, 응답성 저하

### 6.2 해결책

**적응적 배치 크기 (Adaptive Batch Size)**:

1. 시스템 부하에 따라 배치 크기를 동적으로 조절
2. 낮은 부하 → 작은 배치 (빠른 응답)
3. 높은 부하 → 큰 배치 (높은 처리량)

### 6.3 설계

```cpp
class AdaptiveBatchController {
    size_t currentBatch = 32;
    size_t minBatch = 8;
    size_t maxBatch = 256;

    void adjust() {
        double load = getMailboxLoad();
        if (load > 0.7) {
            currentBatch = std::min(currentBatch * 2, maxBatch);
        } else if (load < 0.2) {
            currentBatch = std::max(currentBatch / 2, minBatch);
        }
    }
};
```

### 6.4 예상 효과

- **균형잡힌 성능**: 부하 수준에 따른 최적 배치
- **지연 시간 개선**: 낮은 부하 시 빠른 응답
- **처리량 유지**: 높은 부하 시 효율적 배치

## 7. 개선된 메트릭 시스템

### 7.1 문제

현재 메트릭 시스템의 문제점:
- 측정 중에도 메트릭 수집이 활성화되면 오버헤드 발생
- 뮤텍스 기반 카운터가 경쟁 유발
- 메트릭 데이터가 주기적으로 저장되지 않음

### 7.2 해결책

**잠금 해제 메트릭 시스템**:

1. **원자적 카운터**: `std::atomic<int64_t>`로 뮤텍스 제거
2. **배치 수집**: 주기적으로 메트릭을 수집하여 저장
3. ** 선택적 활성화**: 벤치마크 중 자동 비활성화

### 7.3 설계

```cpp
class LockFreeMetrics {
    std::atomic<int64_t> messagesSent{0};
    std::atomic<int64_t> messagesProcessed{0};
    std::atomic<int64_t> totalLatencyNs{0};

    void recordSend() {
        messagesSent.fetch_add(1, std::memory_order_relaxed);
    }

    void recordProcess(uint64_t latencyNs) {
        messagesProcessed.fetch_add(1, std::memory_order_relaxed);
        totalLatencyNs.fetch_add(latencyNs, std::memory_order_relaxed);
    }

    MetricsSnapshot snapshot() {
        return {
            .sent = messagesSent.load(std::memory_order_relaxed),
            .processed = messagesProcessed.load(std::memory_order_relaxed),
            .avgLatencyNs = totalLatencyNs.load(std::memory_order_relaxed)
                          / std::max(1L, messagesProcessed.load(std::memory_order_relaxed))
        };
    }
};
```

### 7.4 예상 효과

- **측정 오버헤드 90% 이상 감소**
- **동시성 안전성** 보장
- **실시간 모니터링** 가능

## 8. 메시지 라우팅 최적화

### 8.1 문제

현재 메시지 라우팅의 비효율:
- `enqueue()`에서 `push()` 후 `count()` 호출로 **이중 뮤텍스 획득**
- 핫 패스에서 `Time::now()` 호출로 **~12ms overhead/100K**
- `actor_context.cpp:37`의 while 루프에서 **매 메시지 뮤텍스 획득**

### 8.2 해결책

**이중 뮤텍스 제거**:

```cpp
// 현재 (비효율):
void enqueue(Message msg) {
    mailbox_.push(std::move(msg));    // 뮤텍스 1
    auto count = mailbox_.count();    // 뮤텍스 2 (불필요)
    metrics.recordEnqueue(count);
}

// 개선:
void enqueue(Message msg) {
    bool ok = mailbox_.push(std::move(msg));
    if (ok) {
        metrics.recordEnqueue();  // 카운터만 증가
    }
}
```

**배치 팝 최적화**:

```cpp
// 현재 (비효율):
while (mailbox_.pop(msg)) {  // 매번 뮤텍스 획득
    actor_->handle(msg);
}

// 개선: 배치로 한번에 팝
auto batch = mailbox_.popBatch(maxBatch);
for (auto& msg : batch) {
    actor_->handle(msg);
}
```

### 8.3 Time::now() 최적화

```cpp
// 현재 (오버헤드):
auto start = Time::now();
// ... 처리 ...
auto end = Time::now();

// 개선: 타임스탬프 전달
uint64_t start = Time::nowNs();
// ... 처리 ...
// 핸들러에서 타임스탬프 사용
```

### 8.4 예상 효과

- **이중 뮤텍스 제거**: enqueue 오버헤드 50% 감소
- **배치 팝**: 뮤텍스 획득 횟수 maxBatch배 감소
- **Time::now() 최적화**: 핫 패스 오버헤드 제거

## 9. 기타 개선 사항

.large 메시지의 경우 압축 후 전송:

- 재시도 큐 별도 관리

### 9.4 연결 풀링

서비스 액터 간 연결 재사용:
- IPC 연결 풀
- D-Bus 연결 풀

## 10. 구현 우선순위

### 10.1 치명적 (반드시 구현)

| 구분 | 항목 | 예상 기간 |

### 10.2 높음 (강력 권장)

| 구분 | 항목 | 예상 기간 |
|------|------|----------|
| 스케줄링 | 작업 도둑질 스케줄러 | 2-3주 |
| 라우팅 | 배치 팝 최적화 | 1주 |
