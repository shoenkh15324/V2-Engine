# 스케일링 벤치마크

## 목적

> **연구 질문:** "워커나 액터를 추가하면 성능이 선형적으로 향상되는가?"

**워커 및 액터 스케일링 효율성**을 측정합니다. 이상적인 병렬 시스템에서는 2x 워커 = 2x 쓰루풋이지만, 현실은 다릅니다.

## 설계 원칙

### 왜 중요한가

스케일링은 하드웨어 투자가 보상받을 수 있는지를 결정합니다. 코어를 추가해도 도움이 되지 않으면 시스템은 동기화 바인드입니다.

### 측정 방법

**테스트 1: 워커 스케일링** (액터 고정)
```
각 worker_count in [1, 2, 4, 8]에 대해:
    throughput = measure(worker_count, 고정된 actors, iterations)
```

**테스트 2: 액터 스케일링** (워커 고정)
```
각 actor_count in [1, 2, 4, 8]에 대해:
    throughput = measure(고정된 workers, actor_count, iterations)
```

**효율성 계산:**
```
efficiency(N) = throughput(N) / (N × baseThroughput)
```

### 파라미터 참조

| 파라미터 | 기본값 | 설명 |
|---------|--------|------|
| `--workers` | 변동 | 워커 스레드 수 |
| `--actors` | 변동 | 대상 액터 수 |
| `--iterations` | 100,000 | 전체 메시지 수 |
| `--maxbatch` | 32 | 워커당 최대 배치 크기 |
| `--warmup` | 0 | 웜업 메시지 수 |

## 기술적 상세

### 단일 워커가 가장 빠른 이유

- 세마포어 경쟁 없음 (단일 워커가 모든 웨이크업 처리)
- 뮤텍스 경쟁 없음 (하나의 워커만 pop)
- 최대 캐시 로컬리티 (하나의 스레드가 모든 데이터 처리)
- 컨텍스트 스위칭 없음 (스케줄러가 하나의 워커 관리)

### workers=1에서 액터 스케일링이 작동하는 이유

단일 워커가 액터를 순차적으로 처리합니다:
1. 메인 스레드가 액터 A의 메일박스에 push
2. 워커가 액터 A에서 pop → handle
3. 메인 스레드가 액터 B의 메일박스에 push
4. 워커가 액터 B에서 pop → handle

**각 메일박스의 뮤텍스를 하나의 워커만 사용** → 경쟁 없음. 액터가 많아져도 쓰루풋 유지.

### workers>1에서 액터 스케일링이 실패하는 이유

여러 워커가 뮤텍스 경쟁을 발생시킵니다:
1. 워커 1이 액터 A에서 pop을 시도
2. 워커 2가 액터 A에서 pop을 시도 → **뮤텍스 경쟁**
3. 워커 3이 액터 B에서 pop을 시도
4. 워커 4가 액터 B에서 pop을 시도 → **뮤텍스 경쟁**

**여러 워커가 같은 메일박스 뮤텍스를 경쟁** → 쓰루풋 감소.

### 동기화 비용 분석

```
세마포어 acquire/release: ~200-500ns
  └→ 워커 웨이크업 + 컨텍스트 스위칭
뮤텍스 lock/unlock: ~20-50ns
  └→ Mailbox::push/pop
Dispatch (readyQueue push): ~20-50ns

총 동기화 비용: ~300-600ns/메시지
Handle 비용: ~5ns (원자적 카운터)

동기화/handle 비율: 60~120x
```

## 분석

### 핵심 발견

1. **피크 쓰루풋: workers=1, actors=1에서 16.9M msgs/sec**
   - 락프리 push가 단일 워커 시나리오에서 잘 스케일

2. **가벼운 핸들러에서 워커 스케일링 실패**
   - 1→4 워커: 쓰루풋 급감 (16.9M→25K)
   - 동기화 비용 > 병렬 이점
   - **핸들러 무게에 맞춰 워커 수 결정**

3. **단일 워커에서만 액터 스케일링 작동**
   - workers=1: 액터 수에 따라 쓰루풋 감소
   - workers=4: 쓰루풋 안정적이지만 낮음
   - **단일 워커가 스케일링에 최적**

4. **LockFreeMailbox 최적: 단일 워커 + 단일 액터**
   - 16.9M msgs/sec 달성 가능
   - **이상을 초과하면 성능 크게 저하**

### 권장사항

| 우선순위 | 권장사항 | 예상 효과 |
|---------|---------|----------|
| 높음 | 피크 쓰루풋을 위해 workers=1 사용 | 16.9M msgs/sec |
| 중간 | 핸들러 무게에 맞춰 워커 수 결정 | 병렬 활용 |
| 중간 | 최대 쓰루풋을 위해 단일 액터 사용 | 16.9M msgs/sec |
| 낮음 | 단일 워커에서만 액터 수 증가 | 쓰루풋 유지 |

## 결론

### 요약

가벼운 핸들러 시나리오에서 워커 스케일링은 실패합니다 — 동기화 비용(300~600ns)이 핸들러 비용(5ns)을 압도합니다. LockFreeMailbox는 workers=1, actors=1에서 16.9M msgs/sec를 달성하지만, 단일 워커를 초과하면 성능이 크게 저하됩니다.

### 사용 시점

| 시나리오 | 권장사항 |
|---------|---------|
| 단일 워커, 높은 쓰루풋 | actors=1, workers=1 |
| 무거운 핸들러 (I/O, DB) | CPU 코어 수에 맞춰 워커 스케일 |
| 많은 액터, 단일 워커 | LockFreeMailbox (최적) |
