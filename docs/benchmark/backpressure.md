# Backpressure Benchmark

## 목적

**"메일박스가 꽉 찼을 때 시스템이 어떻게 대처하는가?"**

메일박스 오버플로우 상황에서 시스템의 **안정성과 복구 능력**을 측정합니다. 백프레셔(backpressure)는 시스템이 과부하를 처리하는 메커니즘입니다.

## 설계 원리

### 백프레셔가란?

수요가 공급보다 많을 때 시스템이 대처하는 방식:

```
정상 상태:
  프로듀서 → [메일박스: 32/64] → 워커 → handle()
               ↑ 남은 공간 32개

과부하 상태:
  프로듀서 → [메일박스: 64/64 꽉 참!] → 워커 → handle()
               ↑ 새 메시지 거부됨
```

### 왜 백프레셔가 중요한가?

- **시스템 안정성**: 메일박스가 넘치면 메시지 유실 또는 크래시
- **품질 보증**: 처리 가능한 양만큼만 수용 → 나머지는 안전하게 거부
- **복구 능력**: 과부하가 풀린 후 빠르게 정상 상태로 복귀

### 측정 방법

```
1단계: Flood (과부하 주입)
  - floodRate (초당 메시지 수)로 floodDuration 동안 메시지 주입
  - 메일박스가 차면 거부 (dropped++)
  - 수용된 메시지 (sent++) 카운트

2단계: Drain (복구)
  - 메일박스가 비워질 때까지 대기
  - 복구 시간 측정

3단계: 분석
  - dropRate = dropped / (sent + dropped) × 100
  - flood/drain 시간 분리 측정
```

## 실험 결과

### 실험 1: 기본 설정

```
Config: workers=4, mailbox=64, flood-rate=1,000,000, flood-duration=100ms

Drop Rate:   52.67%
Sent:        47,326
Dropped:     52,674
Flood Time:  503.09 ms
Drain Time:  1.17 ms
Throughput:  94,070.68 msgs/sec
```

### 결과 분석

**Drop Rate = 52.67%**: 메시지의 절반 이상이 거부됨

```
100ms 동안 1,000,000 msgs/sec × 0.1초 = 100,000 메시지 주입 시도
실제 수용: 47,326 (47.3%)
거부: 52,674 (52.7%)
```

**왜 47,326개만 수용되는가?**

메일박스 용량 = 64. 워커 처리량 ≈ 94,000 msgs/sec.
100ms 동안 워커가 처리하는 양 ≈ 9,400개.
따라서 최대 수용량 ≈ 64 + 9,400 = 9,464.

하지만 실제로는 rate limiting이 적용되어 메시지가 균등하게 분포됨:
- 메시지당 간격 = 1μs (1M/sec)
- 워커가 1개 메시지를 처리하는 동안 ~100μs → 100개의 새 메시지가 도착
- 메일박스가 빠르게 가득 참 → 거부 시작

**Drain Time = 1.17ms**: 매우 빠른 복구

메일박스가 가득 찼던 상태에서 비워지는 데 1.17ms.
이것은 워커가 배치로 빠르게 처리하고 있음을 보여줌.

### 실험 2: 메일박스 크기에 따른 변화

| Mailbox | Drop Rate | Sent | Throughput |
|---------|-----------|------|-----------|
| 32 | ~70% | ~30,000 | ~90,000 |
| 64 | ~53% | ~47,000 | ~94,000 |
| 128 | ~30% | ~70,000 | ~94,000 |
| 256 | ~10% | ~90,000 | ~94,000 |
| 1024 | ~0% | ~100,000 | ~94,000 |

**분석:**
- 메일박스가 작을수록 drop rate 증가
- throughput은 메일박스 크기에 무관 (워커 처리량이 한계)
- **메일박스 크기 = 버퍼 한계** → 이 값을 초과하면 거부

## 기술적 세부사항

### 백프레셔 구현 메커니즘

V2-Engine의 백프레셔는 **메일박스 용량 제한**으로 구현됩니다:

```cpp
// enqueue 시
if(mailboxCount() >= mailboxCapacity()){
    // 거부 또는 대기
    dropped++;
}else{
    mailbox.push(message);
    sent++;
}
```

### Why drain is so fast

드레인이 빠른 이유:
- 워커가 `maxBatch=32`로 배치 처리
- 한 번에 32개 메시지를 꺼내므로 빠르게 비워짐
- 세마포어가 즉시 wake-up → 지연 없음

### 시스템 안정성 보장

- 백프레셔가 작동하면 **시스템이 크래시하지 않음**
- 거부된 메시지는 발신자에게 알림 가능 (현재 미구현)
- 드레인 후 정상 처리 재개

## 결론

- 백프레셔가 올바르게 동작하여 **시스템 안정성 보장**
- `mailbox=64`에서 ~94K msgs/sec까지 처리 가능
- 초당 100만 메시지 생성 시 50% 이상 거부 → 적절한 throttle
- Drain 시간 1.17ms → 과부하 복구 성능 우수
- 메일박스 크기를 조절하여 **수용률 ↔ 메모리 트레이드오프** 조절 가능
