# 백프레셔 벤치마크

## 목적

> **연구 질문:** "메일박스가 넘칠 때 시스템이 어떻게 동작하는가?"

**메일박스 오버플로우 하에서의 안정성 및 복구**를 측정합니다. 백프레셔는 프로듀서가 컨슈머보다 빠를 때 시스템 크래시를 방지합니다.

## 설계 원칙

### 왜 중요한가

백프레셔가 없으면 가득 찬 메일박스는 크래시나 무한 메모리 성장을 초래합니다. V2-Engine은 오버플로우 시 **조용히 드롭**하는 백프레셔를 구현합니다.

### 측정 방법

세 가지 테스트 케이스로 서로 다른 오버플로우 시나리오를 측정합니다:

| TC | 이름 | 설명 | 핵심 지표 |
|----|------|------|----------|
| 1 | 빈 상태 시작 | 워커가 먼저 시작된 후 플러드 | 드롭율 |
| 2 | 사전 채우기 | 메일박스를 채운 후 워커 시작 | 드레인 시간 |
| 3 | 거의 찬 상태 | 80% 사전 채우기 + 시작 + 플러드 | 드롭율 + 드레인 시간 |

```
1. Flood:    플러드 속도(100K 메시지, 100ms)까지 메시지 push
2. Drain:    processed == sent until (busy-wait)
3. Analyze:  drop_rate, drain_time
```

### 파라미터 참조

| 파라미터 | 기본값 | 설명 |
|---------|--------|------|
| `--workers` | 변동 | 워커 스레드 수 |
| `--maxbatch` | 변동 | 워커당 최대 배치 크기 |

| `flood-rate` | 100,000 | 시도할 메시지 수 |
| `flood-duration` | 100 | 플러드 시간 (ms) |

## 기술적 상세

### 백프레셔 구현

V2-Engine은 메일박스 가득 참 시 **조용히 드롭**합니다:

```cpp
// Mailbox::push
template <typename U>
bool push(U&& msg){
    std::lock_guard<std::mutex> lock(mutex_);
    if(count_ == capacity_){
        return false;              // 조용히 드롭
    }
    buffer_[tail_] = std::forward<U>(msg);
    tail_ = (tail_ + 1) % capacity_;
    ++count_;
    return true;
}
```

**핵심 특징:**
- **조용히 드롭**: `sendMsg()`가 `void` 반환 — 발신자가 드롭 감지 불가
- **조건**: `count_ == capacity_` (가득 참만, 하이워터 마크 없음)
- **메트릭만**: `Metrics::recordEnqueue()`가 드롭 수 추적

### 드레인 메커니즘

드레인은 **세마포어를 통한 협력적 스케줄링**을 사용합니다:

```cpp
// 워커 루프
void Worker::runLoop(){
    while(running_){
        ActorContext* actorCtx = dispatcher_->acquire();   // 세마포어 차단
        if(!actorCtx) break;
        int processed = actorCtx->run(maxBatch_);          // 배치 처리
    }
}
```

**드레인 흐름:**
1. `dispatch()` → readyQueue에 추가 + `sema_.release()`로 워커 웨이크업
2. `Worker::acquire()` → 워커가 세마포어에서 깨어남
3. `actorCtx->run(maxBatch)` → 최대 maxBatch개 메시지 처리
4. 메일박스가 비지 않으면 → `dispatch()` 다시 → 다음 배치

### 시스템 안정성

- 백프레셔가 오버플로우 시 크래시를 방지
- 드롭된 메시지에 대해 발신자에게 알릴 수 있음 (현재 미구현)
- 드레인 후 정상 동작으로 복구

## 분석

### 핵심 발견

1. **maxBatch가 드롭율 제어**
   - maxBatch=1: ~86% 드롭
   - maxBatch=32: **0% 드롭**
   - **maxBatch를 늘려 드롭 제거**

2. **메일박스 크기가 드레인 시간에 비례적 영향**
   - 메일박스=64: 드레인 ~0.02ms
   - 메일박스=8192: 드레인 ~2.3ms
   - **용량과 복구 속도의 트레이드오프**

3. **워커 수가 드롭율에 미치는 영향은 미미**
   - 플러드 시간은 워커 수 증가에 따라 증가 (뮤텍스 경쟁)
   - maxBatch=32에서 드롭율은 안정적
   - **워커는 주입 속도에 영향, 오버플로우 동작에 영향 없음**

4. **세 가지 테스트 케이스로 포괄적 커버리지**
   - TC1: 처리 중 플러드 성능
   - TC2: 순수 드레인 시간 측정
   - TC3: 혼합 시나리오

### 권장사항

| 우선순위 | 권장사항 | 예상 효과 |
|---------|---------|----------|
| 높음 | 0% 드롭을 위해 maxBatch >= 32 설정 | 드롭 제거 |
| 중간 | 빠른 드레인을 위해 메일박스 작게 유지 | 빠른 복구 |
| 중간 | 드레인 벤치마크를 위해 TC2 사용 | 격리된 측정 |
| 낮음 | 워커는 플러드 시간에 영향, 드롭율에 영향 없음 | 쓰루풋 최적화 |

## 결론

### 요약

백프레셔는 maxBatch >= 32에서 0% 드롭으로 올바르게 작동합니다. 메일박스 크기가 드레인 시간에 비례적 영향을 미칩니다. 워커 수가 드롭율에 미치는 영향은 미미합니다.

### 사용 시점

| 시나리오 | 권장사항 |
|---------|---------|
| 0% 드롭 필수 | maxBatch >= 32 |
| 빠른 드레인 필요 | 작은 메일박스 (64~256), 큰 maxBatch |
| 안정성 우선 | 조용히 드롭 메커니즘이 크래시 방지 |
| 혼합 부하 | TC3 시나리오: 80% 사전 채우기 + 플러드 |
