| 항목 | 값 |
|------|-----|

## 벤치마크별 심층 분석

### 1. Throughput: 시스템의 처리 능력

**결론**: 동기화 인프라만으로 초당 ~100K 메시지 처리 가능

**의미**:
- 1개 메시지당 ~10μs 소요 (enqueue + dispatch + handle)
- 뮤텍스 + 세마포어의 오버헤드가 이 수준
- 실제 핸들러가 I/O(1ms)를 수행하면 ~1,000 msgs/sec 예상

**설계 시사점**:
- 핸들러가 가벼운 시스템(카운터, 상태 추적)에 적합
- 무거운 핸들러(DB, 파일 I/O)는 별도 최적화 필요

### 2. Latency: 실시간성 검증

**결론**: P50=6.2μs, P99=28.4μs → 실시간 시스템에 적합

**의미**:
- 메시지의 절반은 6.2μs 안에 처리
- 99%는 28.4μs 안에 처리
- P99와 P50의 차이(4.5배)는 배치 처리 때문

**Tail Latency의 중요성**:
```
P50 = 6.2μs  → "대부분 빠름"
P99 = 28.4μs → "100번 중 1번은 느림"
Max = 321μs  → "최악의 경우"
```

실시간 오디오에서 321μs 지연은 audibly noticeable할 수 있음.
하지만 일반적인 메시징 시스템에서는 충분히 빠름.

### 3. Contention: 동시성 처리 능력

**결론**: 8개 프로듀서에서 ~470K msgs/sec → 뮤텍스 경쟁에 강함

**의미**:
- 뮤텍스의 critical section이 극히 짧음 (~50-100ns)
- 8개 스레드가 동시에 push해도 포화되지 않음
- 초당 100만 메시지 생성 시스템에도 충분

**뮤트екс 오버헤드 추정**:
```
단일 스레드 push: ~120K msgs/sec → push당 ~8.3μs
8 프로듀서:       ~470K msgs/sec → push당 ~2.1μs (스레드당)
→ 뮤텍스 오버헤드: ~50-100ns
```

### 4. Scaling: 확장성 한계

**결론**: 경량 핸들러에서는 **싱글 워커가 최적**

**왜 이런 결과가 나오는가?**
```
핸들링 비용:  ~5ns  (원자적 증가)
동기화 비용: ~500ns (뮤텍스 + 세마포어 + 컨텍스트 스위칭)
→ 동기화 비용이 핸들링 비용의 100배
```

**개선 방향**:
1. 핸들러에 실제 작업(파일 I/O, DB 등) 추가 → 동기화 비용 상쇄
2. Lock-free 메일박스 도입 → 뮤텍스 경쟁 제거
3. 배치 크기 최적화 → dispatch 빈도 감소

### 5. Backpressure: 안정성 검증

**결론**: 백프레셔가 올바르게 동작하여 시스템 안정성 보장

**의미**:
- `mailbox=64`에서 ~94K msgs/sec까지 처리 가능
- 초당 100만 메시지 생성 시 53% 거부 → 적절한 throttle
- Drain 시간 1.17ms → 과부하 복구 성능 우수

**시스템 안전성**:
```
과부하 시:
  프로듀서 → [메일박스 꽉 참] → 거부 (drop rate 53%)
               ↓
  워커 → 빠르게 처리 → 메일박스 비워짐 (1.17ms)
               ↓
  정상 상태 복귀
```

### 6. Scheduler: 시간 정밀도 검증

**결론**: 드리프트 -0.02%, 지터 ±3ms → 우수한 정밀도

**의미**:
- 커널 타이머(timerfd) + epoll로 충분한 정밀도 달성
- ±3ms 지터는 일반적인 백그라운드 작업에 적합
- 발사 누락 없음 → 타이머 신뢰성 보장

**적용 가능 영역**:
- heartbeat (1초 간격) → 충분
- 정기 동기화 (10초 간격) → 충분
- 게임 tick (16.6ms) → ±3ms 지터는 부적합 (별도 최적화 필요)

## 시스템 전체 평가

### 강점 (Strengths)

1. **낮은 레이턴시**: P50=6.2μs는 상당히 빠름
2. **높은 동시성 처리**: 8 프로듀서에서 ~470K msgs/sec
3. **안정적인 백프레셔**: 과부하 시 안전하게 throttle
4. **정확한 타이머**: 0.02% 드리프트
5. **간단한 아키텍처**: 뮤텍스 기반으로 구현 용이

### 한계 (Limitations)

1. **스케일링 부재**: 경량 핸들러에서 워커 증가 시 성능 저하
2. **배치 처리 트레이드오프**: maxBatch 증가 시 P99 레이턴시 증가
3. **뮤트екс 기반**: Lock-free 구조 대비 동시성 한계
4. **지터**: ±3ms는 실시간 오디오/비디오에는 부족

### 개선 방안

| 우선순위 | 개선 사항 | 기대 효과 |
|---------|----------|----------|
| 높음 | Lock-free 메일박스 도입 | 스케일링 개선 |
| 높음 | maxBatch 동적 조절 | 레이턴시/처리량 최적화 |
| 중간 | 핸들러 무게 프로파일링 | 최적 워커 수 결정 |
| 중간 | P99 레이턴시 모니터링 | 실시간 안정성 추적 |
| 낮음 | 하드웨어 타이머 활용 | 지터 개선 |

## 학술적 기여

이 벤치마크 결과는 다음을 증명합니다:

1. **뮤텍스 기반 메일박스의 성능 특성**: 동시 push에 강하지만 스케일링에 취약
2. **배치 처리의 트레이드오프**: throughput 향상 vs tail latency 증가
3. **경량 핸들러에서의 병렬화 한계**: 동기화 비용이 핸들링 비용을 초과하면 스케일링 무효
4. **백프레셔 메커니즘의 실효성**: 메일박스 용량 제한으로 안정성 확보
5. **커널 타이머의 정밀도**: timerfd + epoll로 ±3ms 지터 달성
