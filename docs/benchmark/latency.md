# 레이턴시 벤치마크

## 목적

> **연구 질문:** "단일 메시지가 주입부터 처리까지 얼마나 걸리는가?"

**단일 메시지의 종단간 레이턴시**를 측정합니다. 실시간 시스템에서는 쓰루풋보다 응답 시간이 더 중요합니다.

## 설계 원칙

### 왜 중요한가

쓰루풋은 "얼마나 빠른지"를 측정하지만, 레이턴시는 **"메시지가 실제로 처리되기까지 얼마나 걸리는지"**를 측정합니다. 게임 서버, 실시간 채팅, IoT에서 가장 중요한 지표입니다.

### 측정 방법

```
각 메시지마다:
  1. sendTimestamp[i] = now()           ← 주입 시점
  2. receiveMsg(Tick{})                 ← 메일박스에 push
  3. while not handled: spin-wait       ← handle() 대기
  4. latency[i] = now() - sendTimestamp ← handle()에서 측정
```

- **동기식**: 한 번에 하나의 메시지, 배치 없음
- **동기화 오버헤드**만 측정 (세마포어 웨이크업 + 뮤텍스)

### 파라미터 참조

| 파라미터 | 기본값 | 설명 |
|---------|--------|------|
| `--workers` | 4 | 워커 스레드 수 |
| `--iterations` | 50,000 | 측정할 메시지 수 |
| `--maxbatch` | 32 | 워커당 최대 배치 크기 |
| `--warmup` | 0 | 웜업 메시지 수 |
| `--mailbox` | 0 | 메일박스 용량 (0=자동) |

## 실험 결과

### 기본 파라미터

| 파라미터 | 값 | 설명 |
|---------|-----|------|
| iterations | 50,000 | 전체 메시지 수 |
| warmup | 1,000 | 웜업 메시지 수 |

### 실험 1: 워커 수

**조건:** maxBatch=32, iterations=50,000

| Workers | P50 | P95 | P99 | P999 | Max |
|---------|-----|-----|-----|------|-----|
| 1 | **378 ns** | 472 ns | **641 ns** | 15.3μs | 232μs |
| 2 | 440 ns | 3.4μs | 17.2μs | 35.0μs | 169μs |
| 4 | 10.9μs | 14.5μs | 21.9μs | 44.1μs | 247μs |
| 8 | 11.1μs | 14.5μs | 21.6μs | 54.1μs | 335μs |

**분석:**
- **workers=1: 최저 레이턴시** — P50=378ns, P99=641ns
- **workers=4+: P50 크게 증가** — MPSC 제약 + readyQueue 병목
- **P95~P99은 workers=4+에서 유사** — 시스템 오버헤드 지배

### 실험 2: maxBatch

**조건:** workers=4, iterations=50,000

| maxBatch | P50 | P95 | P99 | P999 | Max |
|----------|-----|-----|-----|------|-----|
| 1 | 10.7μs | 14.5μs | 27.6μs | 46.5μs | 227μs |
| 4 | 11.6μs | 14.5μs | 26.7μs | 46.5μs | 227μs |
| 32 | 10.9μs | 14.5μs | 21.9μs | 46.5μs | 227μs |
| 64 | 10.9μs | 14.5μs | 27.2μs | 46.5μs | 227μs |
| 128 | 11.3μs | 14.5μs | 27.0μs | 46.5μs | 227μs |

**분석:**
- **P50은 maxBatch에 무관하게 안정적** — ~10.7~11.6μs
- **P95~P99 고정** — ~15μs/~27μs (시스템 오버헤드)
- **핵심:** workers=4에서 maxBatch가 레이턴시에 미치는 영향은 미미

### 실험 3: 반복 횟수

**조건:** workers=4, maxBatch=32

| 반복 횟수 | 평균 레이턴시 | P50 | P95 | P99 | P999 | Max | 쓰루풋 |
|----------|-------------|-----|-----|-----|------|-----|--------|
| 10,000 | 5,526 ns | 1,531 ns | 14,799 ns | 27,429 ns | 61,327 ns | 172,022 ns | 46,451 |
| 50,000 | 5,769 ns | 1,761 ns | 14,799 ns | 27,139 ns | 46,529 ns | 227,444 ns | 43,611 |
| 100,000 | 5,763 ns | 1,754 ns | 14,820 ns | 27,388 ns | 49,235 ns | 261,341 ns | 43,971 |

**분석:**
- **반복 횟수에 관계없이 레이턴시 분포 안정적** — P50~P99 거의 동일
- **Max는 반복 횟수 증가에 따라 증가** — 100k에서 시스템 부하 아웃라이어
- **핵심:** 반복 횟수에 관계없이 결과가 **안정적**

### 결과 요약

| 변수 | P50 변화 | P95~P99 변화 | 쓰루풋 변화 |
|------|---------|-------------|------------|
| workers ↑ | **증가** (378ns→11.1μs) | **증가** (472ns→14.5μs) | **감소** |
| maxBatch ↑ | **무관** (~11μs) | **무관** (~15μs/~27μs) | **약간 증가** |
| iterations ↑ | **무관** (~1.7μs) | **무관** (~15μs/~27μs) | **무관** (~44K) |

## 기술적 상세

### 레이턴시 구성 요소

```
레이턴시 = 세마포어 대기 + 뮤텍스 잠금 + pop + handle + ack

세마포어:  sema_.acquire()          ~200-1,000 ns (워커 수에 따라 변동)
뮤텍스:    std::lock_guard          ~20-50 ns
Pop:       Mailbox::pop()           ~50-100 ns
Handle:    counter_.fetch_add(1)    ~5-10 ns
Ack:       ackCounter_.fetch_add(1) ~5-10 ns
──────────────────────────────────────────────
총 (workers=1):                     ~350-500 ns
총 (workers=4):                     ~1,500-5,000 ns
```

### workers=1이 가장 빠른 이유

동기식 측정에서는 메일박스에 하나의 메시지만 존재합니다:
1. 메인 스레드 push → dispatch → sema_.release()
2. 워커 sema_.acquire() → pop() → handle() → ack
3. 메인 스레드 ack 확인 → 다음 메시지

**더 많은 워커는:**
- sema_.release()에서 썬더링 허드
- pop()에서 뮤텍스 경쟁
- 스케줄링 레이턴시 증가

### P99 > P50인 이유

P99 아웃라이어는 다음에서 발생:
1. 세마포어 웨이크업 변동
2. 다른 스레드와의 컨텍스트 스위칭
3. 시스템 부하 (OS 스케줄러, 캐시 미스)
4. 인터럽트/GC

## 분석

### 핵심 발견

1. **workers=1에서 최저 레이턴시**: P50=378ns, P99=641ns
   - 이론적 최소값에 근접
   - **실시간 시스템에 적합**

2. **workers=4+에서 레이턴시 증가**
   - P50이 ~11μs로 증가
   - MPSC 제약 + readyQueue 병목

3. **maxBatch가 레이턴시에 미치는 영향은 미미**
   - P95~P99이 ~15μs/~27μs로 고정
   - **테일 레이턴시는 환경에 의존**

4. **반복 횟수에 관계없이 결과 안정적**
   - 10k~100k 반복에서 동일한 분포
   - **높은 측정 신뢰도**

### 권장사항

| 우선순위 | 권장사항 | 예상 효과 |
|---------|---------|----------|
| 높음 | 최저 레이턴시를 위해 workers=1 사용 | P50=378ns, P99=641ns |
| 중간 | 반복 횟수 >= 10,000 유지 | 통계적 신뢰도 |
| 낮음 | maxBatch가 레이턴시에 미치는 영향은 미미 | 쓰루풋 최적화에 집중 |

## 결론

### 요약

시스템은 workers=1에서 **P50=378ns, P99=641ns**를 달성합니다. workers=4+에서 MPSC 제약로 인해 레이턴시가 크게 증가합니다. 테일 레이턴시(P95~P99)는 maxBatch나 반복 횟수에 관계없이 안정적입니다.

### 사용 시점

| 시나리오 | 권장사항 |
|---------|---------|
| 실시간 오디오/비디오, HFT | workers=1, maxBatch=32 |
| 게임 서버, 실시간 채팅 | workers=2~4, maxBatch=32 |
| 일반 웹 서비스 | workers=4~8, maxBatch=32~64 |
