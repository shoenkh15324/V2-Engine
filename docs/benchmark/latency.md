# Latency Benchmark

## 목적

**"메시지 하나가 들어오고 실제로 처리될 때까지 얼마나 걸리는가?"**

개별 메시지의 **끝단 레이턴시(end-to-end latency)**를 측정합니다. 실시간 시스템에서 가장 중요한 지표 중 하나입니다.

## 설계 원리

### 레이턴시가 중요한 이유

Throughput이 "얼마나 빨리 처리하는가"라면, Latency는 **"메시지 하나가 도착한 후 실제로 처리될 때까지의 대기 시간"**입니다:

```
시점 A: Main Thread에서 receiveMsg(Tick{}) 호출 → enqueue → push → dispatch
시점 B: Worker 스레드에서 Actor::handle() 실행 → latency = B - A
```

- 게임 서버: 플레이어 입력 → 응답까지의 시간
- 실시간 채팅: 메시지 전송 → 수신까지의 시간
- IoT: 센서 데이터 → 처리까지의 시간

### 동기식 측정 방식

이 벤치마크는 **동기식(synchronous)**으로 측정합니다:

```
for each message:
  1. sendTimestamp[i] = now()           ← 시점 A
  2. receiveMsg(Tick{})                 ← 메시지 주입
  3. while not handled: spin-wait       ← 처리될 때까지 대기
  4. latency[i] = now() - sendTimestamp ← 시점 B (actor 내부에서 계산)
```

- 메시지 1개를 보내고 **반드시 처리된 후** 다음 메시지를 보냄
- mailbox에 메시지가 1개만 존재하므로 batching 효과 없음
- **동기화 오버헤드(세마포어 wake-up + 뮤텍스)만을 순수하게 측정**

### 레이턴시와 처리량의 관계

```
Throughput = 전체 메시지 수 / 전체 시간
Latency = 개별 메시지 대기 시간
```

**동시에 최적화하기 어렵습니다.** 배치 처리(batch processing)는 throughput을 높이지만 latency의 꼬리(tail)를 길게 만듭니다.

### 레이턴시 분포

단순 평균만으로는 불완전합니다. **퍼센타일(percentile)** 분포가 필요합니다:

| 퍼센타일 | 의미 | 예시 |
|---------|------|------|
| **P50** | 절반의 메시지가 이 시간 안에 처리됨 | "보통 이 정도 걸림" |
| **P95** | 95%의 메시지가 이 시간 안에 처리됨 | "거의 대부분 이 정도" |
| **P99** | 99%의 메시지가 이 시간 안에 처리됨 | "100번 중 1번은 이 정도" |
| **P999** | 99.9%의 메시지가 이 시간 안에 처리됨 | "극단적 케이스" |
| **Max** | 가장 느린 메시지 | "최악의 경우" |

### 핵심 파라미터

| 파라미터 | 기본값 | 의미 |
|---------|-------|------|
| `--workers` | 4 | 워커 스레드 수 |
| `--iterations` | 100,000 | 측정할 메시지 수 |
| `--maxbatch` | 32 | 워커당 최대 배치 크기 |
| `--warmup` | 0 | 워밍업 메시지 수 |
| `--mailbox` | 자동 | 메일박스 용량 (0=iterations+256) |

**분석:**
- **workers=1에서 최저 레이턴시**: P50=357ns, P99=881ns — 거의 순수 세마포어 wake-up 비용
- **workers가 많을수록 레이턴시 증가**: 세마포어/뮤텍스 경쟁으로 인한 워커 스케줄링 지연
- **workers=1의 P99=881ns**: 이론적 최저 레이턴시 — 세마포어 acquire + 뮤텍스 lock/unlock + handle
- **throughput도 workers=1에서 최고**: 레이턴시가 낮으면 throughput도 높음 (동기식 측정이므로)
- **핵심**: 동기식 측정에서는 **워커가 많을수록 불리** — 경쟁 오버헤드만 증가

## 기술적 세부사항

### 레이턴시 구성 요소

```
레이턴시 = 세마포어 웨이트 + 뮤텍스 lock + pop + handle + ack

세마포어 웨이트:   sema_.acquire()          ~200-1,000 ns (workers 수에 따라 변동)
뮤텍스 lock:      std::lock_guard          ~20-50 ns
pop:              Mailbox::pop()            ~50-100 ns (카운트 감소 + 포인터 갱신)
handle:           counter_.fetch_add(1)     ~5-10 ns (원자적 연산)
ack:              ackCounter_.fetch_add(1)  ~5-10 ns
───────────────────────────────────────────────
총합 (workers=1):                          ~350-500 ns
총합 (workers=4):                          ~1,500-5,000 ns
```

### Why workers=1이 가장 빠른가?

동기식 측정에서는 메시지가 mailbox에 1개만 존재합니다:
1. Main Thread가 push() → dispatch() → sema_.release()
2. Worker가 sema_.acquire() → pop() → handle() → ack
3. Main Thread가 ack 확인 → 다음 메시지 전송

**workers가 많으면:**
- 세마포어 release 시 여러 워커가 동시에 acquire 시도 → **thundering herd**
- 뮤텍스 pop() 경쟁 → 컨텍스트 스위칭 비용
- 실행 가능한 워커가 많을수록 스케줄링 지연 증가

### Why P99가 P50보다 높은가?

배치 처리의 기작:

```
Mailbox: [M1][M2][M3]...[M32] (32개 메시지)
                ↓
Worker:  M1, M2, M3, ..., M32를 한 번에 꺼내 처리
```

하지만 이 벤치마크는 **동기식**이므로 mailbox에 메시지가 1개만 있습니다. P99의 원인은:

1. **세마포어 wake-up 지연**: 워커가 sleep에서 깨어나는 시간 변동
2. **컨텍스트 스위칭**: 다른 워커/스레드와의 CPU 경쟁
3. **시스템 부하**: OS 스케줄러, 캐시 미스 등
4. **GC/인터럽트**: 시스템 레벨의 간헐적 지연

## 결론

### 핵심 발견

1. **workers=1에서 최저 레이턴시: P50=357ns, P99=881ns**
   - 거의 순수 세마포어 wake-up + 뮤텍스 비용
   - 이론적 최저 레이턴시에 근접

2. **workers가 많을수록 레이턴시 증가**
   - workers=4: P50=469ns, P99=18.5μs
   - workers=8: P50=1,068ns, P99=28.4μs
   - **동기식 측정에서는 워커 경쟁이 독이 됨**

3. **maxBatch는 P50에만 영향, tail latency는 무관**
   - maxBatch=64~128에서 P50=825~965ns로 개선
   - P95~P99는 ~15μs/~27μs로 고정 — 시스템 오버헤드가 지배

4. **레이턴시 분포가 iterations에 관계없이 안정적**
   - 10k~100k 반복에서 P50~P99가 거의 동일
   - **측정 신뢰도 높음**

5. **Throughput과 레이턴시의 역설**
   - workers=1: 레이턴시 최저(357ns), throughput 최고(190만)
   - workers=8: 레이턴시 최고(4.7μs), throughput 최저(2.4만)
   - **동기식 측정에서는 레이턴시와 throughput이 동일한 방향으로 움직임**

6. **실시간 시스템 적합성 판단**
   - workers=1: P50=357ns, P99=881ns → **오디오/비디오, 고빈도 트레이딩에 적합**
   - workers=4: P50=469ns, P99=18.5μs → **게임 서버, 실시간 채팅에 적합**
   - workers=8: P50=1.1μs, P99=28.4μs → **일반 웹 서비스에 적합**

### 레이턴시 가이드

| 레이턴시 | 용도 적합성 | 권장 설정 |
|---------|-----------|----------|
| P50 < 500ns | 실시간 오디오/비디오, 고빈도 트레이딩 | workers=1, maxBatch=32 |
| P50 < 2μs | 게임 서버, 실시간 채팅 | workers=2~4, maxBatch=32 |
| P50 < 10μs | 일반적인 웹 서비스 | workers=4~8, maxBatch=32~64 |
| P99 > 10ms | 실시간 앱에서는 부적합 | 워커 수/배치 크기 조절 필요 |

### 권장 사항

1. **최저 레이턴시가 필요하면 workers=1 사용**
   - P50=357ns, P99=881ns 달성 가능
   - 단, throughput은 단일 워커에 의존

2. **maxBatch=64~128에서 P50 개선**
   - 세마포어 wake-up 후 더 많은 메시지를 한 번에 처리
   - 단, tail latency(P95~P99)는 개선 없음

3. **Tail latency 제어가 중요하면 workers를 적게**
   - workers=4: P99=18.5μs
   - workers=8: P99=28.4μs (+54%)
   - **워커 수는 tail latency에 직접적 영향**
